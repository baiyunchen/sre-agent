# 项目概述
一个人工智能的Agent框架, 用来帮助团队做自动化运维。

# 项目背景
团队每天需要有专人花一定的时间用来处理生产环境的alert，但很多alert的处理流程只是follow特定的流程进行检查和处理，如果能使用LLM的能力将这部分工作尽可能的自动化，只在必要的时候引入人工的确认，那将会极大程度上减轻团队的工作负担。

在实现上，我们希望实现借鉴和参考人类工程师来处理Alert的方式，让Agent自行调用各种工具，来进行故障定位，并给出修复建议，暂时不考虑让Agent自行修复故障。如果Agent受限于某些权限或者工具时，可以求助于人类工程师来给输入上下文。Agent在使用工具时，需要人类工程师进行批准，但需要能记住某些工具的审批结果（永久允许或永久禁用）。AI Agent整个的处理过程，应该对人类完全透明化，需要将所有做决策的细节都可视化给人类。


# 团队现状
1. 针对很多的alert，团队已有Confluence文档，来记录当某个告警发生时，应该遵循咋样的流程来判断当前告警是什么原因导致的，这个confluence文档可以覆盖80%左右的场景
2. 对于已有的Playbook没有覆盖的场景，团队的开发需要自行结合代码来判断发生的原因
3. 团队会运维多个服务，不同服务使用的技术栈、架构、部署平台可能都有差异
4. 团队的日志主要存储在Sumo Logic和Cloud watch中，大部分系统集成了OpenTelemetry用于做Trace，系统的Metrics主要使用Prometheus+Grafana和AWS的Metris工具来收集
5. 团队的服务主要部署在AWS和公司自建的K8S集群中，代码使用Github来进行托管，使用Buildkite作为部署流水线。团队主要使用AWS Cloudwatch和Prometheus的Alert来触发告警，目前使用Slack来接受告警消息


# 技术架构
## 技术栈
- 使用.NET 10 + C#
- 大模型主要使用Open AI API兼容的供应商
- 不使用Microsoft Agent Framework这种重量级框架，仅使用Microsoft.Extension.AI这种轻量级的封装
- 使用AWS Knowledge Base来做向量存储和检索
- 使用Postgres作为数据库
- 使用EF Core作为ORM框架

## 架构分层 
架构设计需要完全解耦AI Agent框架和SRE的具体需求，需要能让AI Agent的框架能适配于将来更多的需求。

### AI Agent框架的技术要求
- 使用多Agent架构来帮助AI Agent精简上下文
- 多Agent之间使用文件来共享上下文并精简上下文
- 使用Skill这种渐进式上下文披露计数来优化上下文长度
- 使用事件总线的方式来传递AI Agent执行过程中发生的事件
- 需要很好的管理上下文，避免上下文溢出，在上下文超过特定长度时，自动进行裁剪
- 工具调用的审批需要做统一封装，避免代码重复调用
- AI Agent的执行过程需要持久化，记录所有的细节，并且需要能从中断中回复和人工中断、人工干预等
- AI Agent执行的上下文消息要支持定制化的裁剪逻辑，但真正交给AI 大模型处理的需要做过滤。所以消息中需要加入类似于Metadata或者Tag这种东西，方便后面做上下文优化
