# 大型变更任务看板

> 用于维护较大需求的拆分任务、开发进度、测试覆盖率与 e2e 结果。  
> 每次进入大型改动前先更新本文件，再开始编码。

最后更新: 2026-03-11

## 使用约定

- 任务优先使用 User Story：作为 `<角色>`，我希望 `<能力>`，以便 `<价值>`
- 任务状态：`todo` / `in_progress` / `blocked` / `done`
- 同一时间仅一个主任务处于 `in_progress`
- 每个任务必须包含：实现、单元测试、集成测试、覆盖率、e2e 结果
- 每次最小可验证改动后进行一次小步 Git 提交，并记录 commit hash
- 角色文档化：产品/架构/QA 必须有文档产物（开发文档可选）
- 每个阶段必须由项目经理评审并给出 `GO/NO-GO`，记录在 `docs/pm/stage-gate-<feature-key>.md`

### US-001 - 会话列表 API（分页/筛选/搜索/排序）

- User Story: 作为 SRE 工程师，我希望在 Sessions 页面看到分页、可筛选、可搜索的会话列表，以便快速定位待处理告警会话
- 状态: `done`
- 负责人: `agent`
- 依赖: 无

#### Acceptance Criteria (Given / When / Then)

- Given 数据库中存在 15 条 Session，When `GET /api/sessions?page=1&pageSize=10`，Then 返回 200，`items` 为 10 条，`total=15`
- Given 存在不同状态和来源的 Session，When `GET /api/sessions?status=Running&source=CloudWatch`，Then 仅返回匹配项
- Given 数据库为空，When `GET /api/sessions`，Then 返回 200 且 `items=[]`、`total=0`
- Given 非法分页参数，When `GET /api/sessions?page=-1` 或 `pageSize=0`，Then 返回 400 和错误信息

#### Contract-first 要求

- 新增 `GET /api/sessions` 契约
- 新增 `SessionSummary`、`SessionListResponse` schema
